## 1. 引言

### 1.1 编写目的

本文档用于统一产品、设计、HarmonyOS 客户端、服务端、AI、管理后台、测试和部署工作的需求边界。所有 P0 功能开发、接口设计、测试用例和竞赛演示均应能追溯到本文档中的需求编号。

### 1.2 产品背景

高校中的竞赛、科研、实践和创新项目机会分散在官网、公众号、群聊、海报和文件中。学生不仅难以快速判断自己是否符合条件，还常在知识准备、资料查找、成员招募、任务分工和过程协作环节中断。与此同时，校内技术分享、课程资料、经验和问答长期散落在群聊与网盘，难以检索、验证和持续维护。学生组织和指导老师开展活动时，也缺少统一的人员、岗位、班次和现场执行工具。

校汇通将机会判断、校内知识、结构化协作和项目/活动执行连接为一条完整履约链路，同时确保 AI 结论可解释、可修改、可追溯且不替代人工决策。

### 1.3 产品目标

1. 将非结构化机会通知转化为带来源的结构化机会对象。
2. 帮助个人或已有团队逐条判断资格、缺口、风险和下一步行动。
3. 让组队成为可选流程，并通过结构化岗位提高匹配和确认效率。
4. 建立按学科、主题和内容类型组织的校园知识与资源社区。
5. 为长期项目和固定时间活动提供不同但互通的执行工具。
6. 通过 HarmonyOS 多设备协同降低捕获、规划和现场执行之间的切换成本。

## 2. 总体说明

### 2.1 产品范围

系统包括 HarmonyOS 用户端、管理 Web 端、业务 API、实时消息与异步任务、AI 网关、关系数据库、向量检索、对象存储、缓存队列、日志监控和部署脚本。

核心业务域如下：

```text
身份与组织
  ├─ 可行：机会采集 -> AI 解析 -> 可行性报告 -> 行动路线
  ├─ 知享：分类内容 -> 问答互动 -> 资源包 -> 质量治理
  ├─ 协作：岗位发布 -> 申请/邀请 -> 队长审批/候选人确认 -> 团队确认
  └─ 项目执行
      ├─ 项目：里程碑 -> 任务 -> 交付 -> 复盘
      └─ 活动：岗位 -> 班次 -> 现场执行 -> 归档
```

### 2.2 设计原则

- 可行性判断与组队解耦，组队与项目创建解耦。
- 申请不直接产生成员关系：队伍申请由队长批准，活动报名由负责老师批准；邀请或指定还需学生本人确认。
- 协作需求创建者在正式团队成立前拥有“临时队长/招募负责人”权限；正式团队成立后该身份转为队长，或由其授权移交给另一名正式成员。
- 学科分类、内容类型、资源类型、技能标签和可见范围分别建模。
- 项目与活动分别建模，仅复用通用成员、任务、文件、评论和通知能力。
- AI 输出是建议或草稿，关键业务动作必须由用户确认。
- 私有报告、项目文件和组织数据不进入公共搜索和公共推荐。
- 所有关键结论、审核和权限变更可追溯。

### 2.3 用户角色

| 角色 | 主要目标 | 核心权限 |
| --- | --- | --- |
| 游客 | 了解公开内容 | 浏览允许公开的机会和社区内容，不可互动 |
| 学生 | 判断机会、学习分享、组队和执行 | 报告、发帖、资源、申请、项目、活动参与 |
| 队长（团队内角色） | 维护当前队伍 | 查看授权证据，批准/拒绝/候补本队申请，发送邀请 |
| 教师/导师 | 指导项目、组织活动 | 项目点评、交付确认、活动负责、报名最终审批 |
| 组织负责人 | 协调社团/学生会活动 | 在负责老师监管下发布、初筛、排班和复盘 |
| 内容管理员 | 维护社区质量 | 分类、资源审核、举报和版权处理 |
| 平台管理员 | 维护平台与权限 | 用户、组织、角色、机会、AI 配置和审计 |

### 2.4 运行环境

- P0 首发终端为 HarmonyOS 手机、平板和 PC/2in1，并根据窗口宽度切换底部导航或侧栏工作区。
- 穿戴端为 P2 独立模块，不阻塞 P0/P1 主应用上线。
- 管理端支持当前稳定版 Chromium 内核浏览器。
- 生产服务部署于华为云，开发和演示环境可使用 Docker Compose。

### 2.5 假设与依赖

- P0 支持后台人工认证教师/组织负责人并提供演示身份，不依赖真实教务或学校统一身份系统。
- OCR、大模型和 Embedding 通过可替换 Provider 接入。
- 外部模型不可用时，系统仍应允许人工录入、修改和继续执行。
- 校内官方身份、课程、场地、设备和志愿时长系统在 P2 通过适配器接入。

## 3. 业务流程需求

### 3.1 个人机会履约

```text
录入机会 -> 识别与核对 -> 生成个人报告 -> 查看依据与风险
-> 获取知识/资料建议 -> 生成行动清单 -> 标记报名状态 -> 归档结果
```

### 3.2 组队机会履约

```text
个人报告 -> 选择发起组队 -> 建立结构化岗位 -> 候选人申请/队长邀请
-> 加入草稿团队 -> 临时队长确认正式成立 -> 队长创建项目
-> 项目成员逐一接受加入 -> 生成任务草案并由 ProjectOwner 确认
-> 分工执行 -> 交付 -> 复盘
```

### 3.3 校园知识共享

```text
选择主分类和内容类型 -> 编辑帖子或上传资源包
-> 帖子自动安全检查 / 资源文件扫描与人工审核
-> 分类页展示 -> 检索/互动/下载 -> 纠错与版本维护
-> 经授权被机会报告或项目任务引用
```

### 3.4 校园活动协作

```text
授权主体创建活动并绑定负责老师 -> 老师接受责任 -> 活动审批 -> 创建岗位/班次 -> 指定学生或开放招募
-> 老师批准报名/学生确认指定 -> 流程与任务下发 -> 现场状态与异常
-> 出勤/贡献确认 -> 复盘归档
```

## 4. 功能需求

需求优先级含义：`P0` 必须完成；`P1` 公测前完成；`P2` 规模化阶段完成。

| --- | --- | --- | --- |

| FR-ID-002 | P0 | 系统应支持维护昵称、头像、学校、学院、专业、年级和联系方式可见性 | 修改后多端一致，隐私字段不公开泄露 |
| FR-ID-003 | P0 | 系统应实现学生、教师、组织负责人、内容管理员和平台管理员角色 | 无权限角色访问受限接口返回 403 |
| FR-ID-004 | P0 | 系统应支持基础教师和组织负责人认证；P1 再接入学校统一身份认证 | P0 支持人工/后台认证、撤销和到期；P1 支持学校身份源同步 |

| 编号 | 优先级 | 需求描述 | 验收要点 |
| --- | --- | --- | --- |
| FR-CAP-001 | P0 | 用户应能维护技能、兴趣、可投入时间和组队意愿 | 每个字段可单独配置可见范围 |

| FR-OPP-002 | P0 | 系统应保存原始文件、官方链接、采集时间和内容指纹 | 用户可从结构化字段回看原始依据 |

| FR-OPP-005 | P0 | 用户应能修改解析字段，系统应保留 AI 值、用户值和版本 | 修改后报告使用确认值，审计可回看原值 |
| FR-OPP-006 | P1 | 系统应识别重复机会和多个通知版本 | 用户可合并，冲突字段必须并列确认 |

### 4.5 可行性判断与行动路线

| 编号 | 优先级 | 需求描述 | 验收要点 |
| --- | --- | --- | --- |
| FR-FEA-001 | P0 | 系统应支持个人可行性报告 | 报告只使用用户授权资料和已确认机会字段 |
| FR-FEA-002 | P0 | 系统应支持已有团队整体可行性报告 | 团队报告不覆盖成员个人报告，且只使用成员已授权的证据范围 |
| FR-FEA-003 | P0 | 每条条件应标记符合、不符合、缺少、风险、信息不足/无法判断或待确认 | “信息不足/无法判断”用于资料未授权或证据不足，不得当作“不符合”；不使用无法解释的单一“成功率”代替逐条结论 |
| FR-FEA-004 | P0 | 每条结论应显示机会侧依据、用户侧依据、置信度和核验建议 | 无依据时必须标记无法判断 |
| FR-FEA-005 | P0 | 系统应生成材料清单、关键日期、前置条件和下一步行动 | 行动可加入个人待办并追踪完成状态 |
| FR-FEA-006 | P0 | 系统应根据知识缺口推荐对应分类的社区内容 | 只推荐已公开且未下架内容，并显示普通用户、教师推荐、已验证或官方来源等级；社区内容不得参与官方资格判定 |
| FR-FEA-007 | P0 | 系统可建议团队缺口岗位，但不得自动发起招募 | 用户确认后才生成协作需求草稿 |
| FR-FEA-008 | P0 | 报告应支持重新生成、人工修正和归档 | 版本间可比较，不覆盖历史报告 |
| FR-FEA-009 | P0 | 外部模型失败时应允许重试或转人工填写 | 失败不得导致已录入原始资料丢失 |
| FR-FEA-010 | P0 | 团队成员应能单独授权、查看和撤回团队报告对其能力证据的使用范围 | 未授权部分显示“信息不足/无法判断”而非“不符合”；撤回后新报告不得使用，历史报告标记权限已变化 |

### 4.6 协作广场与组队

| 编号 | 优先级 | 需求描述 | 验收要点 |
| --- | --- | --- | --- |
| FR-COL-001 | P0 | 系统应支持机会组队、普通项目招募和活动岗位招募三种协作需求 | 列表和详情明确显示类型，不混用申请流程 |
| FR-COL-002 | P0 | 组队岗位应包含职责、技能、证据、人数、投入时间、截止和可见范围 | 缺少必填字段不能发布 |

### 5.3 数据保留与删除

- 用户删除的公开内容先软删除，保留申诉和审计所需记录。
- 文件对象使用引用计数，仍被有效版本引用时不得物理删除。
- 用户注销后，按法规和平台规则匿名化公开贡献，删除或隔离个人敏感数据。
- AI 原始输入、输出和反馈按数据分类设置保留期限；私有材料不得用于模型训练，除非用户明确授权。

## 8. 验收场景

### AC-01 机会解析与个人判断

学生在手机端上传竞赛海报，系统提取名称、资格、队伍人数、截止和材料。学生修改一个低置信字段后生成报告；报告逐条显示原文、用户资料依据、状态和行动项，并可在平板接续查看。

### AC-02 只判断、不组队

学生完成个人报告后选择个人参赛。系统生成个人行动清单和社区学习推荐，不创建协作需求、团队或项目。

### AC-03 组队并转项目

学生从报告中的能力缺口生成两个岗位，创建者成为临时队长并作为草稿团队首个成员。候选人申请并授权展示证据，只有临时队长批准申请后候选人才加入草稿团队；另一名候选人通过队长邀请并由本人接受后加入。临时队长手动确认正式成立后转为队长，团队仍可继续招募。队长创建校汇通项目后，正式团队成员逐一确认加入项目，ProjectOwner 再确认任务草案和角色。

### AC-04 分类技术分享

计算机专业学生发布 ArkTS 技术文章，选择“计算机与人工智能 > HarmonyOS”主分类、技术分享类型和 ArkTS 标签。文章通过自动安全检查后标记为用户生成内容并进入分类页；命中风险规则时转人工审核，不进入推荐和语义索引。

### AC-05 资源上传审核

学生上传一个含源码和实验报告的资源包，填写分类、类型、版本和版权声明。文件先通过恶意文件扫描，再由管理员审核；只有扫描安全且审核通过的版本才能预览和下载。发布新版本时旧公开版本继续可用，新版本必须重新扫描和审核。

### AC-06 项目执行

团队项目默认由队长担任 ProjectOwner，手动项目默认由创建者担任。项目成员接受邀请后才能访问项目；ProjectOwner 让 AI 生成任务草案并确认后才正式创建。只有项目管理角色能指派、取消任务和修改依赖，成员按角色提交任务，验收人通过或退回。成员退出前必须完成任务移交，历史交付和评论保留。

### AC-07 教师指定活动人员

组织负责人创建活动并邀请指导老师承担责任，老师接受后活动才能发布。活动中直接指定一名学生、接收一名学生主动报名，并处理一名组织推荐学生；三种路径分别完成确认。一人退出后系统按候补顺序通知替补。

### AC-08 权限与 AI 边界

非项目成员无法访问私有文件；普通学生无法发布官方活动；队伍申请在队长批准前不能进入正式团队；活动报名在负责老师批准前不能进入正式名单；没有原文依据的资格字段显示待确认或无法判断；AI 生成的任务不会在 ProjectOwner 确认前正式指派。

### AC-09 外部服务降级

AI Provider 超时后，系统保留上传文件和 OCR 结果，提示重试或人工填写。用户完成手工录入后仍能继续生成行动清单和创建项目。

### AC-10 名额并发与候补

一个队伍岗位或活动岗位只剩一个名额时，同时批准两名候选人。系统只能批准一人，另一人自动进入候补；空缺后第一候补收到带截止时间的补位邀请，拒绝或超时后自动推进下一候补；名额、通知和审计记录保持一致。

### AC-11 负责人移交与成员隐私

队长发起移交后，新队长接受才生效；负责老师移交同样需要新老师确认。成员撤回团队报告证据授权后，新报告不能继续使用对应证据，成员退出后岗位按候补规则补位。

### AC-12 机会规则更新

管理员确认机会截止或资格字段变化后，旧可行性报告立即标记待复核，行动清单和项目中的导入规则显示差异提示；已经完成或正在执行的任务不被自动覆盖。

### AC-13 团队成立与项目所有权

临时队长在岗位未招满时也能确认团队正式成立并继续招募。创建项目后队长成为 ProjectOwner；手动项目由创建者拥有，所有权移交只有在新负责人接受后生效。

### AC-14 任务依赖与成员退出

系统拒绝形成循环依赖的任务关系并显示冲突路径。成员正常退出前必须移交未完成任务；被紧急移除后 ProjectOwner 重新分配任务，历史文件和贡献仍可审计。

### AC-15 活动责任、截止与取消

组织负责人选择负责老师后，老师接受前活动不能发布、开放报名、审批或指派。报名截止后待审批项自动过期；活动时间、地点或班次变更时，受影响人员必须重新确认。活动进行中可按候补顺序替补空缺岗位，非候补紧急增员必须由老师记录理由。取消活动后关闭报名和签到，并按是否已开始处理实际服务时长。

### AC-16 内容版本与文件安全

项目引用帖子时保存当时的 PostRevision，原作者更新后只提示新版本；已发布帖子编辑后先保留旧公开版本，新的 PostRevision 通过安全检查后才能替换。包含恶意文件的资源包或新 ResourceVersion 被隔离，管理员不能绕过扫描将其公开。

### AC-17 项目成员与外部 AI 数据授权

队长创建项目后，正式团队成员逐一接受项目邀请；未接受者无法访问项目文件。ProjectOwner 确认任务角色和状态权限。需要外部模型处理私有证据时，系统展示数据类别、供应商和留存策略并取得单独授权；未授权时转人工处理。

## 9. 版本范围

## 10. 不在当前范围

- 闲置交易、外卖、跑腿、支付和与机会履约无关的校园生活服务。
- 由平台承担官方报名、官方资格认定或比赛结果预测。
- 使用 AI 自动录取队员、处分用户、决定现场安全措施或代替教师审批。
- P0 直接接入真实教务、场地和设备系统。
- 将实体物资共享与技术知识/文件社区混为同一业务对象。
