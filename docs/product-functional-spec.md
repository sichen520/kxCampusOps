# 校汇通产品功能规格

**文档版本：**V1.6（2026-07-20）

## 1. 产品名称与定位

**完整名称：**《校汇通：基于 HarmonyOS 多设备协同的 AI 高校机会履约与校园协作平台》

**一句话定位：**把分散的高校机会、校内知识和协作需求转化为可解释的判断、可选择的组队方案以及可追踪的项目或活动执行过程。

**核心闭环：**

```text
发现机会 -> AI 解析 -> 可行性判断 -> 补知识/补资料/补队友
-> 建立项目或活动 -> 多端协同执行 -> 交付与复盘 -> 沉淀能力证据和共享知识
```

平台由四个核心域构成：

1. **可行**：回答“这是什么机会、我是否符合、还缺什么、下一步做什么”。
2. **知享社区**：按学科和主题归类校内技术分享、学习资料、问答与经验。
3. **协作广场**：承载结构化的比赛组队、项目招募和活动岗位招募。
4. **项目执行**：分别管理长期项目与固定时间活动的实际执行。

知享社区不是附加论坛，而是平台的知识供给层。可行性报告发现知识缺口后，可以直接推荐对应分类的文章、问答和资料；项目完成后，成员也可以在取得团队授权后把经验沉淀回社区。

## 2. 业务对象与边界

| 对象 | 定义 | 归属模块 | 关键特征 |
| --- | --- | --- | --- |
| 机会 | 可报名、申请或参与的竞赛、科研、实践、基金等信息 | 可行 | 有资格条件、截止时间、材料和官方来源 |
| 可行性报告 | 某个用户或团队针对某个机会的解释性判断 | 可行 | 私有、带原文依据、置信度和待确认项 |
| 协作需求 | 对队友、项目成员或活动人员的结构化招募 | 协作广场 | 有能力/岗位要求、人数、时间和申请状态 |
| 项目 | 围绕成果交付进行的持续性协作 | 项目模式 | 有目标、团队、里程碑、任务和交付物 |
| 活动 | 围绕固定时间、地点和现场岗位进行的组织工作 | 活动模式 | 有主办方、人员、班次、流程和现场状态 |
| 知识帖子 | 技术文章、问答、经验、讨论或项目展示 | 知享社区 | 适合阅读、讨论、收藏和检索 |
| 资源包 | 课件、笔记、试卷、实验报告、源码、电子书等文件集合 | 知享社区 | 有分类、版本、审核、预览和下载记录 |
| 场地/设备/物资 | 项目或活动执行时使用的实体资源 | 项目执行 | 不作为知识论坛内容，后续对接学校资源系统 |

必须遵守以下边界：

- **可行性判断与组队分开。** 用户可以只生成报告和个人行动清单，不被强制组队。
- **组队成功不等于自动建项目。** 发起人确认团队和目标后，才创建校汇通项目空间。
- **申请不等于正式加入。** 队伍申请必须经队长批准，活动报名必须经负责老师批准；队长或老师主动邀请/指定时，也必须由学生本人确认。
- **比赛/项目组队不是普通论坛帖子。** 社区可以展示招募卡片，但数据和申请流程归协作广场。
- **技术求助可以是论坛问答。** 一旦需要明确成员、职责、工期和交付物，应升级为协作需求或项目。
- **活动与项目分开存储和展示。** 二者共用成员、任务、文件、评论和通知基础能力，但流程不同。

## 3. 统一分类体系

平台使用一套共享的“学科-主题-技能”分类体系，连接社区内容、机会要求、能力护照、组队岗位和项目任务。

### 3.1 分类维度

1. **学科分类（内容讲什么）**
   - 一级分类示例：计算机与人工智能、电子与自动化、机械与制造、土木与建筑、艺术与设计、经济与管理、语言与人文、医学与健康、公共基础。
   - 二级分类示例：计算机与人工智能下设 HarmonyOS、前端、后端、人工智能、算法、数据库、网络安全、云计算与运维。
   - 分类由管理员配置，不硬编码在客户端。
2. **内容类型（内容以什么形式出现）**
   - 技术分享、教程、课程答疑、问题求助、竞赛经验、项目复盘、项目展示、讨论。
3. **资源类型（附件是什么）**
   - 课程课件、课堂笔记、历年试卷、实验报告、项目源码、电子书、考研资料、数据集、模板和工具。
4. **标签（具体技术或知识点）**
   - 示例：ArkTS、ArkUI、Spring Boot、Vue、计算机网络、事务、Wireshark。
5. **可见范围（谁能看到）**
   - 全校、指定学院、指定专业、指定课程、指定组织；默认按发布者权限选择。

### 3.2 归类规则

- 每篇帖子或每个资源包必须选择一个主分类，可选一个次分类和多个标签。
- 同一主分类内容进入同一分类页，用户可以继续按内容类型、资源类型、标签和范围筛选。
- AI 只能推荐分类和标签，发布者必须确认；管理员可纠正错误分类。
- 搜索结果必须显示所属分类，避免所有内容混在同一信息流中。

## 4. 用户角色与权限

### 4.1 学生

- 浏览机会、生成个人可行性报告、维护能力护照。
- 发布知识帖子、问题和资源包，参与评论、回答、收藏和评价。
- 发起或申请组队；申请经队长批准后才能成为正式成员并参与项目。
- 报名活动岗位、接收组织者指派、确认或提出冲突；报名经负责老师批准后才能正式参加。
- 只查看自己有权限访问的团队、活动和组织数据。

### 4.2 教师/指导老师

- 拥有学生的全部公开能力，并可作为导师加入项目。
- 学生创建比赛队伍时可以不选择指导老师，也可以发送指导邀请；老师接受前没有该队伍权限，接受后与队长共享成员审批、项目分工、任务发布和验收权限。
- 对项目目标、里程碑和交付物进行确认、点评或退回。
- 在已授权组织下发布活动、指定学生、开放招募、调整岗位和查看执行状态。
- 审批活动岗位报名；只有负责老师批准的报名者才能进入活动正式人员名单。
- 推荐优质社区内容，但不能绕过平台审核规则删除他人内容。

### 4.3 学生组织/社团负责人

- 在所属且已认证的组织内发布活动。
- 在负责老师监管下从组织成员名册推荐学生，或向指定范围开放岗位招募。
- 可以进行报名材料整理和初步筛选，但活动参与者的最终批准权属于负责老师。
- 管理活动岗位、班次、签到、请假、替补、通知和复盘。
- 无权查看组织范围外的隐私数据，也不能获得教师专属审批权限。

### 4.4 平台管理员/内容管理员

- 管理学校、学院、专业、课程、组织、角色和认证状态。
- 管理分类树、标签、社区内容、资源审核、举报、版权和违规处理。
- 管理机会来源、纠错记录、活动发布资格、AI 配置和审计日志。
- 查看脱敏运营数据，不默认读取用户私有可行性报告和项目文件。

## 5. 客户端信息架构

手机端底部导航保持五项：

1. **首页**：待办、截止提醒、进行中的项目/活动、关注分类动态。
2. **可行**：机会发现、扫描解析、可行性报告和个人行动路线。
3. **知享**：分类社区、问答、资源库、关注与搜索。
4. **项目执行**：项目、活动、协作广场三个页签。
5. **我的**：身份、能力护照、发布内容、收藏、下载、积分、通知设置。

平板和 2in1 使用侧边导航，并支持列表-详情分栏、项目看板和活动调度大屏。

## 6. 通用基础功能

### 6.1 账号与身份

- 华为账号登录、受控演示账号、JWT 与刷新令牌会话；不在当前版本承诺手机号/邮箱登录。
- 绑定学校身份，维护学号/工号、学院、专业、年级和所属组织。
- 教师、组织负责人和管理员需要额外认证或后台授权。
- 多设备会话管理、设备下线、账号注销和隐私授权管理。

### 6.2 个人资料与能力护照

- 维护技能、兴趣、可投入时间、求组队状态和可见范围。
- 添加课程项目、竞赛作品、代码仓库、设计稿、报告和交付记录。
- 证据可标记为个人声明、团队确认、教师确认或平台验证。
- 社区优质内容、项目交付和活动贡献只能经用户同意写入能力护照。
- 用户可以隐藏课表、联系方式和私有作品，只公开匹配所需的最少信息。

### 6.3 统一搜索与通知

- 一次搜索机会、社区帖子、资源包和公开协作需求，并按对象类型分组。
- P0 支持基础关键词搜索和分类筛选；P1 增加全文、语义、时间与组合筛选。私有项目和活动数据始终不进入公共搜索。
- 站内信、Push、服务卡片和实况窗按各自阶段承载截止、任务、申请、审核和活动通知。
- 支持按消息类型订阅、免打扰、已读状态和重要提醒升级。

## 7. “可行”机会履约模块

### 7.1 机会采集

- 手机相机扫描海报或二维码。
- 上传截图、PDF、通知文件和宣传图。
- 接收浏览器链接、系统分享、剪贴板文字和手动录入。
- 保存官方来源、采集时间、原文件和内容指纹，识别重复机会。

### 7.2 AI 结构化解析

- OCR 与视觉模型提取机会名称、主办方、赛道、对象、年级/专业限制、队伍人数、截止时间、材料、导师要求、费用和官方链接。
- 解析结果逐字段关联原文位置，低置信字段标记“待确认”。
- 用户可以修改字段并提交纠错，修改记录保留版本和操作者。
- 同一机会的不同通知可以合并，但冲突字段必须并列展示，不能静默覆盖。

### 7.3 可行性判断

- 对照用户身份、能力证据、可投入时间和已有成员逐条判断条件。
- 结果分为“符合”“不符合”“缺少”“风险”“信息不足/无法判断”“待人工确认”，不输出虚假的单一成功率。
- 每条判断显示机会原文依据、用户侧依据、模型置信度和建议核验方式。
- 支持个人判断、已有团队整体判断，两种报告相互独立。
- 团队报告必须先获得每名成员对证据范围的单独授权；成员撤回后，新报告不得继续使用对应证据。
- 用户可选择不提供敏感信息；未授权或缺少信息时，对应条件必须标为“信息不足/无法判断”，不能推断为“不符合”。“待人工确认”用于存在来源但需要权限人员核验的情况。

### 7.4 行动路线

- 生成报名材料清单、关键日期、前置条件和个人下一步行动。
- 根据能力缺口推荐知享社区中的对应教程、问答和资源包。
- 根据团队缺口建议岗位画像，但不自动邀请或自动录取人员。
- 支持把行动项加入日历、服务卡片和个人待办。

### 7.5 机会状态

- 收藏、关注、忽略、准备中、已报名、已结束和已放弃。
- 记录截止变更、规则更新和官方更正，并通知受影响用户；旧报告、行动项和项目导入规则同步标记“待复核”。
- 用户确认差异后生成新版本；系统不得静默覆盖历史判断、已完成任务或正在执行的任务。
- 用户可反馈机会失效、虚假、重复或字段错误，管理员复核后处理。

## 8. 协作广场与可选组队

### 8.1 协作需求类型

- 机会组队：绑定某个比赛、科研或实践机会。
- 普通项目招募：已有明确目标和交付物，但不一定绑定机会。
- 活动岗位招募：绑定某个已发布活动及其岗位/班次。

### 8.2 组队流程

- 发起人从可行性报告一键生成缺口岗位，或手动创建招募需求。
- 发布招募需求时同步创建草稿团队，创建者默认成为临时队长/招募负责人和首个草稿团队成员；创建动作视为本人加入草稿团队的同意。
- 岗位包含能力要求、证据要求、人数、职责、时间投入、截止时间和可见范围。
- 候选人查看匹配理由后主动申请，申请进入“待队长审批”；临时队长/队长批准后，候选人加入草稿团队或已正式成立团队。
- 临时队长/队长也可向候选人发送邀请，邀请必须由候选人接受后才加入团队。
- 支持申请问答、候选对比、队长批准/拒绝/候补和邀请接受/拒绝；主动申请以“学生申请 + 队长批准”达成双向同意，定向邀请以“队长邀请 + 学生接受”达成双向同意。
- AI 根据能力互补、时间重合、证据质量和公开意愿给出可解释推荐，不做最终录取。
- 岗位批准、邀请接受和候补晋级必须锁定剩余名额；并发操作不得超过岗位人数，超出者进入候补。
- 候补名单保存显式顺序，默认按进入候补时间排序；临时队长/队长调整顺序必须填写理由并保留审计。
- 名额空缺后只向当前第一候补发出带截止时间的补位邀请；拒绝或超时后在同一名额事务中推进下一候补，不得重复占用名额。
- 临时队长可在至少一名已完成双向确认的草稿成员存在且岗位未招满时主动确认团队正式成立；确认后草稿成员转为正式成员，临时队长转为队长，团队仍可继续招募，系统不得因满员自动成立团队。
- 成员可主动退出；队长移除成员必须填写原因并通知成员，成员可以申诉。申诉由未参与原决定的平台管理员或预先绑定的团队监督角色复核，不自动恢复身份。
- 成员退出或被移除时撤销后续写权限并保留历史贡献；若已创建项目，还必须完成任务、文件和责任移交。
- 队长可发起权限移交，只有新队长接受后才生效；无队长时冻结招募并由平台管理员指定临时负责人。
- 团队正式成立后，可再次运行团队可行性判断，再由队长创建校汇通项目；项目创建前不存在“授权项目负责人”角色。

### 8.3 活动岗位流程

- 每个正式活动必须绑定一名已接受责任的负责老师；老师本人创建视为接受，学生组织/社团负责人选择老师后必须等待老师确认，确认前不能发布、开放报名、审批报名或指派学生。
- 老师直接指定学生时，老师的批准已成立，但被指定学生仍需确认、请假或提交时间冲突。
- 学生主动报名公开岗位时，状态为“待老师审批”；学生报名本身视为学生同意，老师批准后直接进入正式活动名单。
- 学生组织/社团推荐的学生进入“待老师审批”，老师批准后还需学生确认才正式参加。
- 公开岗位支持老师批准、拒绝、候补和满员自动关闭；AI 和学生组织负责人不能代替老师完成最终批准。
- 活动岗位的主动报名、老师指定和组织推荐分别记录来源审批状态；审批完成后统一进入正式参加、请假、退出、替补、完成或活动取消等履约状态。
- 岗位只统计正式参加者，名额批准和候补晋级必须原子执行。
- 候补名单保存显式顺序，默认按进入候补时间排序；负责老师调整顺序必须填写理由。补位只通知当前第一候补并设置接受截止时间，拒绝或超时后自动推进下一候补。
- 活动必须设置报名截止时间；截止后待审批项自动过期，活动进行中可按候补顺序替补已空缺岗位，禁止普通新增，非候补紧急增员必须由负责老师记录理由。
- 负责老师可以发起责任移交，候选老师接受后生效；参与者批准后仍可退出/请假，空出的名额按候补顺序通知替补。
- 活动时间、地点、岗位或班次变更影响已确认人员时，应展示新旧值并要求受影响人员重新确认；未确认且存在冲突的人员不得被标记为到岗。
- 活动取消后关闭报名、指派、签到和未完成任务并通知相关人员；开始前不产生服务时长，进行中取消则保留实际签到并由负责老师确认有效时长。

## 9. “知享”校园知识与资源社区

该模块吸收原“校园资源共享平台”的资源发布、审核、检索、下载、评价、论坛互动、积分和后台治理能力，并按本项目统一分类体系重新设计。

### 9.1 分类社区首页

- 展示用户关注的学科分类、热门分类、最新内容、优质内容和待回答问题。
- 分类页聚合同一类别下的技术分享、问答、经验和资源包。
- 支持最新、热度、评分、最多收藏、最多下载和未解决排序。
- 用户可关注分类、标签和作者，形成个性化但可切回时间顺序的信息流。

### 9.2 知识帖子

- 发布技术分享、教程、课程答疑、问题求助、竞赛经验、项目复盘、项目展示和讨论。
- 富文本/Markdown 编辑、代码块高亮、公式、图片、视频链接、附件、草稿和预览。
- 发布时选择主分类、内容类型、标签和可见范围。
- AI 提供标题优化、摘要、分类建议、标签建议和相似内容提示，用户确认后才生效。
- 普通帖子发布前先进行自动安全检查；命中风险规则时转人工审核，未通过检查的内容不得进入推荐和语义索引。
- 公开帖子明确标记为用户生成内容，并显示来源等级与最近更新时间，避免与官方规则混淆。
- 已发布帖子每次编辑都创建新的 `PostRevision` 并重新执行安全检查；检查通过前继续展示上一公开版本，风险版本不得进入推荐、索引或替换当前版本。
- 支持编辑历史、撤回、软删除和管理员操作记录；帖子只通过 `currentRevisionId` 指向已通过检查的公开版本。

### 9.3 问答与讨论

- 评论、楼中楼回复、@成员、点赞、收藏、分享和举报。
- 问答帖支持采纳答案、已解决/未解决状态、补充问题和答案排序。
- 技术求助可发布“需要进一步协作”，再转换为结构化项目招募，原帖保留关联入口。
- 帖子不能直接成为项目任务，必须由用户确认引用到具体项目。

### 9.4 资源包

- 上传单文件或多文件资源包，支持断点续传、失败重试和上传进度。
- 填写课程、适用专业、资源类型、版本、学期、说明、标签、来源和版权声明。
- 所有上传文件必须先完成恶意文件扫描；扫描中、扫描失败或发现风险的文件隔离，不能审核通过、预览或下载。
- 通过扫描后支持在线预览常见文档、查看文件清单、版本更新、下载和失效反馈。
- 每个 `ResourceVersion` 独立扫描和审核；新版本待处理时旧公开版本继续可用，风险或未通过版本不得预览、下载或被项目引用。
- 展示浏览量、下载量、收藏量、评分、评论和最近更新时间。
- 首次上传和每次版本更新都进入待审核状态，审核通过后才可成为 `currentVersionId`；审核结果通知发布者。
- 对源码、模板和数据集记录版本与校验值，避免同名文件被静默替换。
- 对外链资源定期检查有效性；失效或存在版权争议时下架但保留审计记录。

### 9.5 检索与推荐

- 按标题、正文、课程、分类、资源类型、标签和作者检索。
- P0 支持基础关键词检索、分类筛选和稳定分页；P1 增加全文检索、语义检索、时间筛选和组合排序。
- 发现近似帖子或重复资源时提示用户先查看已有内容。
- 可行性报告和项目任务可以按技能标签检索社区内容，但只推荐已公开且未下架内容，并标明普通用户、教师推荐、已验证或官方来源等级、更新时间。

### 9.6 质量与激励

- 用户可评分、标记有帮助、举报错误、侵权、恶意文件或失效内容。
- 教师/管理员可标记“教师推荐”“课程适用”“已验证”，并记录推荐人和日期；社区内容不能作为官方资格判断依据。
- 上传审核通过、答案被采纳、内容获认可时记录贡献积分和积分明细。
- 积分只用于贡献展示、徽章或平台权益，不作为能力水平的唯一依据。
- 支持优质作者、优质回答和资源版本维护记录。

## 10. 项目模式

### 10.1 项目建立

- 从已确认团队、某个机会或普通协作需求创建，也可手动创建。
- 定义项目目标、范围、交付物、成员、导师、起止时间和可见范围。
- 每个项目必须有唯一 ProjectOwner；团队项目默认由队长担任，手动项目默认由创建者担任，移交必须由新负责人接受后生效。队长变更不自动改写既有项目所有权，项目必须单独发起所有权移交。
- ProjectOwner 创建项目即成为 `OWNER` 和正式项目成员；团队项目向其他正式团队成员逐一发送项目加入确认，手动项目由 ProjectOwner 邀请。成员接受前状态为“待确认”，不能访问项目文件或任务。
- `ProjectMember` 记录来源、角色、加入时间和正式、退出、移除、冻结状态；角色限 `OWNER`、`MANAGER`、`MEMBER`、`MENTOR`，只有 OWNER 可授予或撤销 MANAGER。团队后续新增成员不会静默加入既有项目。
- 从可行性报告导入规则、材料清单、截止时间和风险，不重复录入；导入项保留报告版本和机会原文引用。
- 机会规则更新后，导入项显示差异和“待复核”，由 ProjectOwner 确认更新，不自动覆盖项目计划和已执行任务。

### 10.2 计划与分工

- 生成 AI 分工前必须补齐项目目标、交付物、里程碑、每位成员的能力证据、经验、可投入时间、任务偏好、限制和任务依赖；缺项时不得生成建议。
- 队长或已接受邀请的指导老师可选择人工分工，也可让 AI 根据完整上下文生成里程碑、任务草案、依赖关系和建议角色。
- ProjectOwner 或已授权项目管理角色确认后才正式创建任务，AI 不自动给成员分配责任；项目内的任务负责人、协作者和验收人只在任务创建后按角色执行。
- 任务包含负责人、协作者、截止时间、优先级、状态、依赖、清单、文件和验收条件。
- 新增或修改依赖时检测有向环；形成循环依赖的操作必须拒绝，并显示冲突路径。
- ProjectOwner 或项目管理角色可指派成员、修改依赖、批准延期、批准转交和取消任务；任务负责人/协作者可开始、提交和标记阻塞；验收人或 ProjectOwner 可通过、退回和重新打开任务。
- 删除任务仅允许未产生交付或审计记录的草稿；其余任务只能取消。所有角色、字段和状态变化写入时间线。
- 支持看板、列表、时间线和个人待办视图。
- 检测任务冲突、无人负责、依赖阻塞和成员负荷不均，并给出可解释建议。

### 10.3 协同执行

- 任务评论、@提醒、文件版本、会议记录、决策记录和变更日志。
- 项目资料可以引用知享社区内容，也可以保存为团队私有知识；引用时锁定 `PostRevision` 或 `ResourceVersion`，新版本只提示比较，不静默替换项目依据。
- 支持成员申请调整截止时间、转交任务和标记阻塞原因。
- 成员正常退出前必须移交未完成任务、文件和责任；紧急移除后由 ProjectOwner 重新分配，历史文件、评论和贡献不得删除。
- 项目附件和交付物同样必须经过文件安全扫描；扫描完成前不得预览或下载。
- 导师可按权限查看里程碑、点评交付物、确认或退回。

### 10.4 风险、交付与复盘

- 根据机会规则和当前进度提示临近截止、材料缺失、关键任务延期和规则变更。
- 交付物按版本提交，保留提交人、时间、审核意见和最终版本。
- 自动汇总周报/阶段报告草稿，所有内容由负责人确认后发送。
- 项目完成后生成时间线、成员贡献、问题与经验复盘。
- 经团队授权后，可把非敏感复盘发布到知享社区；经个人同意后写入能力护照。

## 11. 活动模式

### 11.1 发布与审批

- 教师或已认证组织负责人创建活动，填写主题、主办方、时间、地点、规模、对象和说明；每个正式活动必须绑定已接受责任的负责老师。
- 老师本人创建活动视为接受责任；组织负责人选择老师后进入“待老师确认”，老师接受前不能发布、审批报名、开放岗位或指派学生。
- 支持活动模板，例如讲座、招新、比赛承办、志愿服务、晚会和学院会议。
- 根据学校规则进入教师/组织/平台审批流程，未审批内容不得冒充官方活动。

### 11.2 人员组织

- 创建岗位、职责、人数、能力要求、服务时间、区域和班次。
- 支持学生主动报名、负责老师直接指定、学生组织推荐和候补名单，并分别使用来源审批状态；审批完成后统一进入活动履约状态。
- 自动检测同一学生的班次重叠和已知项目/活动冲突，只提示不强制决定。
- 支持老师批准/拒绝、学生确认、请假、按候补顺序替补、换班和临时增员，并保留变更记录；活动进行中非候补增员必须记录理由。
- 候补名单按显式顺序补位；调整顺序必须填写理由并保留审计。补位邀请包含接受截止时间，拒绝或超时后自动通知下一候补。
- 报名截止后待审批项自动过期；活动进行中可按候补顺序替补已空缺岗位，禁止普通新增，非候补紧急增员必须由负责老师记录理由。
- 活动时间、地点、岗位或班次发生实质变更时，受影响人员进入“待重新确认”，系统保留旧安排和新安排的差异。

### 11.3 活动执行

- 生成活动流程表、岗位任务、物资清单、联系人和关键时间点。
- 手机端查看个人岗位和上报进度；平板端查看人员、区域和任务总览。
- P0 支持基础二维码签到/签退、岗位到位状态、实际服务时长、公告、紧急联系人和事件上报。
- 现场异常由负责人判断和处理，AI 只负责汇总信息、提示冲突和生成备选方案。
- 重要岗位变更实时推送给受影响人员。

### 11.4 结束与归档

- 活动取消时立即关闭报名、指派、签到和未完成任务并通知相关人员；进行中取消保留实际签到记录，由负责老师确认有效时长。
- 汇总出勤、岗位完成情况、异常记录、志愿时长和组织者评价。
- 参与者可对指派、工时和评价提出异议，负责人处理后归档。
- 活动复盘可转为组织私有模板，或脱敏后发布到知享社区。

## 12. AI 能力与安全边界

### 12.1 AI 能力

- 海报、截图、PDF 和网页的 OCR/VLM 机会解析。
- 带原文引用的资格条件抽取和可行性推理。
- P1 提供社区内容分类、标签、摘要、相似内容检测和语义搜索增强。
- 基于公开证据的互补组队推荐与匹配理由。
- 项目任务拆解、进度摘要、风险提示和周报草稿。
- 活动岗位冲突检测、排班建议和现场状态摘要。

### 12.2 强制边界

- AI 不替代官方资格审核，不承诺报名成功或获奖概率。
- AI 不自动录取队员、不自动指定任务负责人、不自动批准活动。
- AI 不把社区内容当作官方规则；官方文件和用户经验必须明确区分。
- 所有关键结论显示来源、时间、置信度和待确认项。
- 模型输出、用户修改和最终决策分别记录，便于追溯和评估。
- 涉及隐私、违规、版权或人员处分时必须由有权限的人处理。
- 私有材料发送给外部模型前，单独展示数据类别、模型供应商、处理区域和留存/删除策略，并取得本次处理授权。
- 仅发送完成任务所需的最少字段；供应商不满足禁训练和留存要求、用户拒绝授权或授权过期时，转人工或本地处理。

## 15. 关键业务闭环

### 15.1 个人机会，不组队

```text
扫描机会 -> AI 解析 -> 个人可行性报告 -> 推荐知识/资料
-> 生成个人行动清单 -> 完成报名 -> 归档结果
```

### 15.2 比赛或项目组队

```text
扫描机会 -> 个人判断 -> 发布结构化岗位 -> 候选申请/队长邀请
-> 加入草稿团队 -> 临时队长确认正式成立 -> 团队可行性报告
-> 队长创建项目 -> 成员逐一确认加入 -> ProjectOwner 确认分工 -> 提交与复盘
```

### 15.3 校园技术分享

```text
选择学科分类 -> 发布技术文章/资源包 -> 自动安全检查/文件扫描 -> 审核
-> 同类内容聚合 -> 搜索/问答/收藏/下载 -> 锁定版本后引用到报告或项目 -> 形成贡献记录
```

### 15.4 教师或社团组织活动

```text
组织创建活动并选择负责老师 -> 老师接受责任 -> 发布/审批 -> 学生报名/老师指定/组织推荐
-> 老师批准报名/学生确认指定/老师批准后学生确认推荐 -> 排班
-> 重大变更重新确认 -> 任务和流程下发 -> 多端签到与现场执行 -> 出勤与异常归档 -> 活动复盘
```

- 基础管理后台：用户、分类、资源审核、举报和机会纠错。

## 17. 技术栈基线

原毕业设计中的 Spring Boot/Vue/MySQL/MinIO 作为功能参考，不直接复制其技术实现。本项目采用统一的可部署技术栈：

### HarmonyOS 客户端

- DevEco Studio、HarmonyOS SDK 6.1.1(24)、ArkTS 严格模式、ArkUI。
- Stage 模型、UIAbility、状态管理 V2、Navigation + NavPathStack。
- MVVM + Repository、relationalStore、Preferences、HTTP/WebSocket。
- Phone、Tablet、PC/2in1 在 P0 首发；Wearable 在 P2 以独立模块接入。
- P0 使用服务卡片、应用接续、Push Kit、Scan Kit；实况窗在 P1 接入。
- Hypium、Hamock、DevEco Testing、Hvigor、ohpm。

### 服务端与实时任务

- Node.js LTS、NestJS、TypeScript、模块化单体架构。
- REST/OpenAPI、Socket.IO WebSocket、class-validator、Pino。
- API 服务与 BullMQ Worker 分进程部署，后续按压力拆服务。
- TypeORM、PostgreSQL 16、pg_trgm、应用侧中文分词、pgvector、Redis、BullMQ。
- 华为账号接入，服务端签发短期 JWT 与可撤销刷新令牌。

### AI 与搜索

- 可插拔 VLM、文本大模型和 Embedding Provider，避免绑定单一模型。
- 华为云 OCR、RAG、pgvector 语义检索、结构化提示词与 JSON Schema 校验。
- 模型调用审计、提示词版本、引用回溯、置信度、失败降级和人工反馈集。

### 文件、后台与部署

- 开发环境 MinIO，生产环境华为云 OBS，文件元数据保存在 PostgreSQL。
- Vue 3、TypeScript、Vite、Pinia、Arco Design 管理后台。
- Docker Compose 用于本地和演示环境。
- 华为云 ECS、RDS、DCS、OBS、SWR 用于首期生产部署；达到规模后再迁移 CCE/Kubernetes。
- TLS、限流、RBAC、对象级权限、审计日志和 ClamAV 文件安全扫描属于 P0；Nginx、生产密钥托管和备份恢复随 P1 生产环境启用。
- 服务端 Jest/Supertest、管理端 Playwright、接口契约测试、迁移测试和部署冒烟测试。

## 18. 当前明确不做

- 不做面向全社会的综合社交平台或校园生活信息大全。
- 不做闲置商品交易、外卖、跑腿等与主闭环无关的功能。
- 不把实体场地/设备共享和知识资源论坛混为一谈。
- 不让 AI 代替官方审核、教师决策、团队录取或现场安全判断。
- P0 不直接接入真实教务系统，先使用可替换的数据接口和演示数据完成闭环。
